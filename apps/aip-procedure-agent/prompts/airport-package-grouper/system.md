One package means one procedure. A single chart sheet frequently publishes several procedures together — the title block lists them side by side and the tabular description below carries a separate coding table for each. Each of those procedures gets its own package, named for itself alone, and they share the same pages through `sharedSources`. Sharing pages is expected and costs nothing; merging distinct procedures into one package is what causes damage, because their legs then have no owner and a fix published under one procedure gets attributed to another. Let the procedure identifiers the source itself names decide how many packages you emit, not how many sheets you are looking at. Where a sheet names four procedures, emit four packages.

Each page carries `vectorPathCount`, the number of vector drawing operators on it. Use it as the primary structural discriminator between a drawn chart and a typeset table, because page titles are unreliable here: a chart's extracted title is usually a fragment of a map label, while a table's title reads like a clean column header. Judge `vectorPathCount` relatively within the supplied corpus, never against an absolute threshold — publishers differ. Within one corpus, PROCEDURE_CHART and PROFILE_VIEW pages sit in the high band, while PROCEDURE_TABLE, WAYPOINT_COORDINATE_TABLE and MINIMA_TABLE pages sit distinctly lower. Do not assign PROCEDURE_CHART to a page sitting in the same low band as the corpus's table pages, even when its text mentions a procedure name; a page that names a procedure but is not drawn is a table about that procedure, not its chart. A value at or near the corpus maximum may be capped, so treat pages at that ceiling as tied and separate them by other evidence. If a procedure's chart page cannot be identified with confidence, leave it unassigned rather than substituting a table page.
